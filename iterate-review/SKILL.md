# iterate-review — GREEN まで回す品質ゲート

「レビュー → 指摘修正 → 再レビュー」を GREEN になるまで繰り返す。1 ラウンド = レビュー 1 回＋修正 1 巡。

## パラメータ

- **対象**: 引数で指定されたファイル / PR / 差分。省略時は現在の作業ツリー差分（`git diff` ＋ untracked）。PR は番号または URL を受け、`gh pr diff` で差分を取得する。対象が特定できない場合は AskUserQuestion で確認する。
- **最大ラウンド数**: `--max-rounds`（既定 5）。上限到達は失敗ではなく「人間の判断が要る」シグナルであり、深く回したい場合のみ明示的に大きい値を指定する。
- **GREEN 条件**: `--green` で指定した severity 以上の指摘がゼロになること（既定 `high`: critical / high がゼロで GREEN。`--green all` は全指摘解消）。

## レビュアーの解決

- セッションに review ロールの配役（例: soloist-codex `--review`）があれば、その配役先がレビュアー。
- 配役が無ければ、Claude サブエージェント（レビュー対象の著者とは別個体）がレビュアー。

1回のループ内で固定するのはレビュアーの**種別**（配役先 / サブエージェント）であり、サブエージェントは毎ラウンド新規起動でよい（却下履歴はレビュー契約が運ぶため、個体の文脈継続は不要）。レビュー実行が失敗した場合（配役先の失敗規律による終了を含む）は、レビュアーを差し替えずにループを停止し、状況・原因・再開方法を報告する。

## レビュー契約（正本）

レビュー1回分の依頼と応答の固定形式。配役スキルはこの契約への準拠を宣言する。

**依頼に必ず含める**:

- レビュー対象の絶対パスと種別（作業差分 / PR 差分 / 文書）。差分の場合は「これは差分ファイルであり、必要に応じてリポジトリ内の関連コードを read-only で調査してよい」と伝える
- 観点: 前提の誤り・検証不足・反証・見落としたエッジケース。read-only で批評せよ、編集・書き込みは禁止
- 指摘形式: `severity(critical|high|medium|low) / 箇所 / 内容 / 推奨修正` で列挙せよ
- 指摘が一件もなければ本文冒頭に `GREEN` とだけ書け
- 前ラウンドで却下した指摘の却下理由（再指摘の防止）

**応答**: 上記形式の指摘列挙、または `GREEN`。

## git 規律

git リポジトリでは、ループは git 履歴に一切触れない。修正は作業ツリー上に累積させ、コミット・stash・ブランチ作成をしない。コミットは GREEN（または妥協終了）後、ユーザーの指示があった場合のみ行う。

## ループ

### Step 1: 対象の確定

レビュー対象を絶対パスで用意する。差分の場合は一時ファイルに書き出し、各ラウンドの開始時に `git diff`（＋未追跡ファイル）を取り直して上書きする。未追跡ファイルは各ファイルを `git diff --no-index /dev/null <ファイル>` で取り、`git diff` の出力に連結する（index に副作用を残さない）。差分は常に HEAD 比なので、修正はコミットしなくても次ラウンドの差分に自動的に反映される。

**PR 対象の場合**: 最初に `gh` の存在と認証を確認する（`command -v gh` と `gh auth status`）。不可なら処理を開始せず、導入・ログイン（`brew install gh` / `gh auth login`）を案内して中止する。差分は初回のみ `gh pr diff` で取得する。修正には PR ブランチのチェックアウトが前提（未チェックアウトなら修正に進まず、Round 1 のレビュー結果を提示して終了する）。修正を行った以降のラウンドは、base ブランチとの差分（`git diff <baseRef>` ＋未追跡ファイル）に切り替えて取り直す（baseRef は `gh pr view <番号または URL> --json baseRefName` で取得する） — `gh pr diff` はローカルの未 push 修正を含まないため。

**git が無い場合**（`git rev-parse --git-dir` が失敗）:

- 対象省略時は差分既定が成立しないため、AskUserQuestion で対象を確認する。ディレクトリ全体を黙って対象にしない。
- ループ開始前に対象ファイルを一時領域へバックアップし、そのパスを報告する（修正の revert 手段の確保）。

完了条件: レビュアーに渡す絶対パスのファイルが存在する。

### Step 2: レビュー実行

レビュー契約の形式で依頼を組み立て、解決済みのレビュアーに渡す。

完了条件: 応答を読み、指摘を severity 別に分類し終えている。

### Step 3: 判定と報告

ラウンドごとに 1 行のスコアボードを報告する: `Round N: critical X / high Y / medium Z / low W（前ラウンドから修正 M 件）`。レビュアー側で再試行が発生したラウンドは、その旨を同じ行に付記する。却下理由・前ラウンドの指摘一覧などラウンド間の記録は一時領域のメモファイルに保持する（リポジトリ内にファイルを作らない）。

- GREEN 条件を満たす → Step 5 へ
- 最大ラウンド到達 → Step 5 へ（妥協終了）
- 同一の指摘（同じ箇所 file:line かつ同趣旨）が 2 ラウンド連続で解消していない → ループを止め、その指摘と対応方針をユーザーに相談する（暴走防止）
- それ以外 → Step 4 へ

完了条件: スコアボードを出力し、上記4分岐のいずれかを選択済み。

### Step 4: 修正

GREEN 条件に関わる指摘（`--green` で指定した severity 以上）は、すべて修正するか却下する。それ未満の指摘は修正せず、最終報告に未対応一覧として記載する（ループの収束を速く保つ）。修正はセッションの現行実装体制に従う（maestro や implement 配役が敷かれていればそれに従い、無ければ Claude が修正する）。指摘が誤り（レビュアーの事実誤認・プロジェクト方針との不一致）と判断した場合は、修正せず却下理由を記録する。修正後、プロジェクトの検証コマンド（test / lint 等）を実行して緑を確認し、Step 1 の対象を更新して Step 2 へ戻る。検証コマンドが該当しない対象（設計書等のドキュメント）では検証をスキップし、最終報告にその旨を明記する。

完了条件: GREEN 条件に関わる全指摘が修正済みか却下記録済みで、検証コマンドが緑（または検証スキップを記録済み）。

### Step 5: 最終報告

以下を含めて報告し、ループを終える:

- 総ラウンド数と最終スコアボード
- GREEN か妥協終了かの別。妥協終了の場合は未解消指摘の一覧と対応方針案（妥協案）
- 却下した指摘と理由
- GREEN 条件未満で未対応の指摘の一覧（あれば）

完了条件: 上記の全項目を出力した。
