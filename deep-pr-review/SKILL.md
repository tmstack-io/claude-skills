# deep-pr-review

GitHub の Pull Request を、通常の `/review` 単体より **多角度かつ高精度** にレビューする。
3 つの観点（標準 PR レビュー / codex 独立レビュー / アーキテクト観点メタ検証）を重ね、最終的に
**重複排除・再ランクした統合レビュー1本** を **正しく分かりやすい日本語** で出力する。

PR への自動コメント投稿は行わない（出力はローカル＝ターミナル表示のみ）。

---

## 入力（引数）

- 引数に PR 番号（例 `1682`）または PR の URL があればそれを対象にする。
- 引数が無ければ、現在のブランチに紐づく PR を `gh pr view --json number,url,title,state,isDraft` で特定する。
  特定できなければ、ユーザーに対象 PR を確認して中止する（推測で別 PR をレビューしない）。

---

## 全体フロー

```
[フェーズ0] 前提チェック & 対象PR特定 & PR head worktree & 共有入力パッケージ作成
        │
        ▼
[フェーズ1] 並列実行
   ├─ A: 標準PRレビュー(独立観点 N 体) → findings_review
   └─ B: codex 独立レビュー(codex exec) → findings_codex   ※codex 無ければ skip
        │
        ▼
[フェーズ2] 機械的マージ & 重複排除（メイン側） → findings_merged
        │
        ▼
[フェーズ3] アーキテクト観点メタ検証（偽陽性除去・抜け漏れ補完・確信度/重要度付与） → findings_verified
        │
        ▼
[フェーズ4] 再ランク（BLOCKER/HIGH/MEDIUM/LOW・全出し・LOW 丸め）
        │
        ▼
[フェーズ5] ドラフト → /clarify-ja で明快化 → 出力 → 後始末
```

最初に TODO リスト（フェーズ0〜5）を作成してから着手すること。

---

## フェーズ0: 前提チェック & 対象PR特定 & 共有入力パッケージ作成

0. **前提スキル clarify-ja の存在チェック**。本スキルはフェーズ5の明快化で
   agent-skills リポジトリの clarify-ja に依存する。次の順で `clarify-ja/SKILL.md` を探す —
   (1) 本スキルの隣（`../clarify-ja/SKILL.md`。両リポジトリのスキルを同じディレクトリへ
   symlink / コピーしていればここで見つかる）、(2) Claude Code のスキルディレクトリの
   プロジェクト側 → グローバル側（`.claude/skills/clarify-ja/` → `~/.claude/skills/clarify-ja/`）。
   どこにも無ければ**レビューを開始せずに中止**し、復旧手順を案内する
   （agent-skills リポジトリ `tmstack-io/agent-skills` から clarify-ja を導入して再実行。
   例: `npx skills add tmstack-io/agent-skills --skill clarify-ja`）。

1. **対象 PR の特定**（上記「入力」参照）。
   `gh pr view <PR> --json number,url,title,state,isDraft,baseRefName,headRefName,headRefOid` で取得。

2. **eligibility チェック**（メイン側で判定。取得済み JSON を見るだけなのでサブエージェントは使わない）。
   次のいずれかなら **中止**し、理由を伝える。
   - PR がクローズ済み / マージ済み。
   - PR がドラフトで、かつユーザーがドラフトのレビューを明示要求していない。
   - PR author が bot アカウント（dependabot / renovate 等の自動生成 PR）で、かつユーザーがレビューを明示要求していない。
   deep-pr-review は深いレビューが目的のため、「既に自分でレビュー済み」では中止しない（再レビューを許容）。

3. **codex 利用可否チェック**
   - `command -v codex` で存在確認。存在すれば `codex --version` で起動可否も軽く確認する。
   - あわせてタイムアウトラッパーを検出する:
     `TIMEOUT_CMD=$(command -v gtimeout || command -v timeout)`（macOS は coreutils の `gtimeout`）。
     未検出でも codex は実行してよい（フェーズ1-B の Bash timeout 指定が上限になる）。
   - codex が無い／実行できない場合は、フェーズ1-B（codex 独立レビュー）を skip する。
     skip した事実は最終出力の `codex:` 行に必ず明記する（codex レビュー済みと詐称しない）。

4. **PR head worktree の作成**
   観点 #1・#3・#4・#5、codex、フェーズ3 の architect は **PR 適用後のファイル・履歴** を参照する。
   現在の作業ツリーが対象 PR と別ブランチだと読む内容がズレるため、PR head を専用の
   git worktree に展開する。作業ツリーは汚さない（現在ブランチを切り替えない）。
   ```sh
   WT_DIR="$(mktemp -d -t deep-pr-review-wt)"
   # pull/<番号>/head は同一リポPR・フォークPRの双方で解決できる。
   # 先頭の + は必須（再実行時に PR head が進んでいても強制更新する。無いと non-fast-forward で失敗する）
   git fetch origin "+pull/<PR番号>/head:refs/deep-pr-review/<PR番号>"
   git worktree add --detach "$WT_DIR" "refs/deep-pr-review/<PR番号>"
   ```
   - **remote 名は PR の取得元に合わせる**: 通常は `origin` でよいが、origin が PR のあるリポジトリを指していない fork 運用（upstream 等）では、`git remote -v` と PR の URL を突合して該当 remote 名に読み替える。
   - **サブエージェントの cwd は指定できない**（セッションの cwd を引き継ぐ）。「$WT_DIR で動かす」は
     プロンプト側で担保する: ファイルを読む各エージェントのプロンプトに `$WT_DIR` の絶対パスを明示し、
     「ファイル参照は `$WT_DIR` 配下の絶対パス、git 操作は `git -C "$WT_DIR"` で行い、
     それ以外の作業ツリーは読まない」ことを指示する。
   - codex のみ `-C "$WT_DIR"` で cwd を直接指定できる（フェーズ1-B）。
   - **フォールバック**: worktree 作成に失敗した場合のみ、作業ツリーがクリーンなことを確認の上
     `gh pr checkout <PR番号>` で切り替え、**終了時に必ず元ブランチへ戻す**。それも不可なら
     「差分のみで継続（ファイル本体は未参照）」とし、その制約を最終出力の `注記:` に明記する。
   - **後始末はフェーズ5参照。正常終了か途中中止かを問わず必須。**

5. **共有入力パッケージの作成**（メイン側。以降のレビューエージェント・codex・architect にはこれを渡す）
   1. **差分ファイル** — 取得は 1 回だけ。以降の全員がこのファイルを参照する:
      ```sh
      DIFF_FILE="$(mktemp -t deep-pr-review-diff)"
      gh pr diff <PR番号> > "$DIFF_FILE"
      ```
   2. **PR 要約** — `gh pr view` の title/body と `$DIFF_FILE` から数行で作成する。
   3. **ルールパス一覧** — `$WT_DIR`（PR head 側）のリポジトリルートの CLAUDE.md / AGENTS.md、
      PR が変更したディレクトリ配下の CLAUDE.md / AGENTS.md、`.claude/rules/` 配下
      （存在すれば。特定プロジェクト名は前提にしない）のパス一覧。
      中身ではなくパス一覧を渡し、各エージェントが必要に応じて自分で読む。
   4. **diff サイズガード** — `wc -l < "$DIFF_FILE"` で規模を確認する。極端に大きい場合
      （目安: 3,000 行超 もしくは 40 ファイル超）は、変更ファイル群を観点ごとに分割するか、
      生成物・lockfile・自動生成コード等を対象外に絞る。
      絞った場合は対象外にしたものを最終出力の `注記:` に明記する（黙って切り捨てない）。

## フェーズ1: 並列実行（A と B を同時に走らせる）

A の N 体（Agent ツール）と B の codex（Bash）は独立なので、**1 メッセージ内で並列起動**する。

### A: 標準 PR レビュー（独立観点 N 体・既定 5 体）

通常の `/review`（組み込み）は結果を PR コメントとして投稿し、構造化された findings が手元に残らない。
そのため **`/review` を呼ばず、その多観点レビューを内包実行**して findings を構造化データとして保持する。

**共通仕様**（全レビューエージェントに適用）:

- `subagent_type: general-purpose`。model は指定しない（セッションの model を引き継ぐ）。
- プロンプトに **読み取り専用**（編集・書き込み・コミット禁止）を明記する。
- 入力: PR 番号 / `$DIFF_FILE` / `$WT_DIR` / PR 要約 / ルールパス一覧 /
  「偽陽性として扱う例」セクション（逐語）。
- 返却フォーマット（逐語で指定する。フェーズ2の機械的マージが崩れないよう厳守させる）:
  ```
  - 観点: #<番号>
    file: <パス>:<行 or 行範囲>
    要約: <一行>
    理由: <なぜ問題か。事実ベースで具体的に>
    推奨修正: <どう直すか>
  ```
  指摘が無ければ「指摘なし」とだけ返させる。

**観点の割り当て**（既定 5 観点）:

- #1: CLAUDE.md / `.claude/rules/` への準拠監査（CLAUDE.md は実装ガイドのため、レビューに無関係な項目は除外する）。
- #2: 変更差分のみを読んだ素朴なバグスキャン（大きなバグ重視。些末な指摘・偽陽性は避ける）。
  この観点のみ `$DIFF_FILE` だけで完結し、`$WT_DIR` を読む必要はない。
- #3: 変更箇所の履歴に照らしたバグ検出。**変更行の由来は base 側で見る**:
  `git -C "$WT_DIR" blame <baseRefName> -- <パス>` や `git -C "$WT_DIR" log -L`。
  （PR head で blame すると変更行は PR 自身のコミットに帰属し、歴史的文脈が読めない。）
- #4: 同じファイルを触った過去 PR のコメントを `gh` API で読み、本 PR にも当てはまる指摘を検出。
  当てはまりの確認には `$WT_DIR` のコードを参照する。
- #5: 変更ファイルのコードコメントを読み（`$WT_DIR` 配下）、コメントのガイダンスに反していないか確認。

**この段階では採点もフィルタもしない。** 確信度はフェーズ3で architect が一元的に付与する
（発見者の自己採点は過大になりやすく、独立した検証者が付けた方が較正される）。
N 体の返却をまとめて `findings_review` とする（各指摘: 出所=review#観点 / file:line / 要約 / 理由 / 推奨修正）。

### B: codex 独立レビュー（`codex exec`）

codex が利用可能な場合のみ実行する（フェーズ0でチェック済み）。

1. 出力ファイルを用意する: `OUT_FILE="$(mktemp -t deep-pr-review-codex)"`
2. **cwd を PR head の worktree にして** read-only で実行する:
   ```sh
   "$TIMEOUT_CMD" --kill-after=10s 300 \
     codex exec --sandbox read-only -C "$WT_DIR" -o "$OUT_FILE" \
     "$DIFF_FILE は GitHub PR の差分です（ベースブランチ: <baseRefName>）。カレントディレクトリは PR 適用後のリポジトリです。この差分を一から独立にコードレビューしてください。バグ・設計上の問題・セキュリティ・整合性・より単純な代替案の観点で指摘し、各指摘に対象ファイルと行(file:line)・問題の理由・推奨修正を付けてください。ただし、PR 変更前から存在する問題、PR が変更していない行への指摘、linter や型チェッカーが拾う類（import 漏れ・型エラー・フォーマット）は除外してください。出力は正しく分かりやすい日本語で、コード識別子や固有名詞は原語のまま記載してください。編集・書き込みは禁止、確認は読み取り専用の調査に限ります。" < /dev/null
   ```
   - `TIMEOUT_CMD` 未検出の環境では wrapper なしで同コマンドを実行する。
   - **Bash ツールの timeout に 360000（6 分）を明示する**。既定の 120 秒では、正常なコールドスタート
     （1〜2 分）＋レビュー時間より先に打ち切られる。
   - `--sandbox read-only` は必須（書込事故防止・非対話での承認待ち回避）。
   - `-o "$OUT_FILE"` は必須。**stdout をパイプで受けず、必ず `$OUT_FILE` を Read する**
     （パイプは無出力ハングに見える）。
   - `< /dev/null` を付け、非対話実行での stdin 待ち・混入を防ぐ。
3. コールドスタートで 1〜2 分は正常。ハング即断・status ポーリングをせず、プロセス終了／
   `$OUT_FILE` 生成を待つ。
4. **タイムアウト到達**（wrapper の終了コード 124 / Bash ツールの timeout / `$OUT_FILE` 未生成 or 空）の場合は、
   `$OUT_FILE` を削除してから**同一コマンドを 1 回だけリトライ**する（初回のみのハングが多い）。
   リトライも失敗したら skip 扱い。
5. `$OUT_FILE` を Read し、指摘を `findings_codex`（各指摘: 出所=codex / file:line / 要約 / 理由 / 推奨修正）として保持する。
6. skip した場合（codex 不在・エラー・2 回タイムアウト）は、最終出力の `codex:` 行に理由付きで明記する。

## フェーズ2: 機械的マージ & 重複排除（メイン側）

`findings_review` と `findings_codex` をマージし、**同一 file:line かつ同趣旨**の指摘を 1 件に統合する。

- 統合時は出所を併記し（例「review#2 / codex」）、最も具体的な説明・推奨修正を採用する。
- 複数出所が同じ問題を挙げた事実は指摘に残す（フェーズ3で確度の補強材料になる）。
- 内容の真偽判断はここではしない（フェーズ3の仕事）。機械的な同一性統合のみ行う。

結果を `findings_merged` とする。重複を先に潰すことで、architect が同じ問題を二重に検証したり、
同一問題への判定が割れたりすることを防ぐ。

---

## フェーズ3: アーキテクト観点メタ検証

`Agent` ツールでサブエージェント（`subagent_type: general-purpose`、`model: opus`）を 1 体起動し、
`findings_merged` をメタ検証させる。

サブエージェントには次のシステムプロンプトでアーキテクト観点を与える。

> あなたは経験豊富なソフトウェアアーキテクト兼コードレビュー監査役である。**読み取り専用**で作業し、いかなるファイルも変更しない（編集・書き込み禁止）。設計の欠陥・抜け漏れ・整合性・依存リスク・より単純な代替案の観点を重視する。

**渡すもの**: `$WT_DIR`（PR 適用後のコードはこの配下を絶対パスで読む）/ `$DIFF_FILE` / PR 要約 /
ルールパス一覧 / `findings_merged` / 「確信度ルーブリック」（逐語）/ 「偽陽性として扱う例」（逐語）。

**求めること**:

- **偽陽性の除去**: 各指摘が実在の問題かを差分と `$WT_DIR` の該当コードに照らして検証し、
  確信度 0–24 相当（明白な偽陽性・既存からの問題・PR 未変更行への指摘）を落とす。
- **抜け漏れの補完**: 既存指摘が見落とした設計上の欠陥・整合性・依存リスク・より単純な代替案を
  新規指摘として追加する（出所=architect）。新規指摘には根拠となる file:line を必須とする。
  出所=architect の指摘は第三者の検証を経ない単独観点である（意図した非対称。読み手は
  出所ラベルで判別できる）。
- **確信度・重要度の付与**: 残った全指摘に、ルーブリックに基づく確信度（0–100）と
  重要度（BLOCKER/HIGH/MEDIUM/LOW）を付ける。

返却フォーマットは、フェーズ1-A の構造に `出所:` `確信度:` `重要度:` を加えたものを逐語で指定する。
結果を `findings_verified` とする。

## フェーズ4: 再ランク（メイン側）

`findings_verified` を次の 4 階層に並べる。**重要度によるフィルタはしない（全出し）**。
ただしフェーズ3で偽陽性と確定したものは出さない。

- **BLOCKER**: 確実なバグ・データ損失・セキュリティ欠陥など、マージ前に必ず直すべき重大問題。
- **HIGH**: 高確信の実バグ。実運用で踏む可能性が高く、現状の実装では不十分。
- **MEDIUM**: 実害はあるが頻度が低い／影響が限定的。
- **LOW**: nitpick・様式・低確信。直すと良いが必須ではない。

各階層内は確信度の高い順に並べる。

- **LOW の肥大化対策**: フィルタはしないが、LOW が多数（目安 8 件超）になった場合は、
  LOW のみ「同種をまとめた要約＋代表 file:line」に丸めてよい。BLOCKER/HIGH/MEDIUM は丸めない。
  丸めた場合は LOW 見出しに「(他 N 件を要約)」と件数を明記する（黙って落とさない）。

---

## フェーズ5: ドラフト → /clarify-ja で明快化 → 出力 → 後始末

統合レビュー1本を内部ドラフトとして作り、`/clarify-ja` スキルの整形規則を適用してから最終出力する。
**PR へのコメント投稿はしない（ターミナル表示のみ）。**

### 1. ドラフト作成

フェーズ4の結果を下記「出力フォーマット」へ流し込み、統合レビューのドラフトを作る。この時点では完璧な日本語にこだわらず、各指摘の「**何が・どこで(file:line)・なぜ問題か・どう直すか**」を漏れなく構造化することを優先する。

### 2. /clarify-ja による明快化（日本語の品質担保）

統合レビューのドラフト全体に `/clarify-ja` の整形規則を適用してから最終出力する。
clarify-ja は本スキルと**同一コンテキストで動く事後リライト規則**であり、別プロセスではない
（Skill ツール等での別呼び出しにはしない）。つまり「渡す」とは、フェーズ0の手順 0 で解決した
`clarify-ja/SKILL.md` を **Read し**、その規則を**この統合レビューに対する最終整形パスとして
同一コンテキストで適用する**ことを指す（意味厳守＝指摘の数・強度・条件・例外・数値・評価語を変えない）。

各指摘の `出所`・`確信度: NN`・重要度見出しとセクション構造は読み手の判断材料であり、
clarify-ja 2-4 の既定どおり保持する（内部タグとみなして削らない）。

### 3. 出力

整形済みの統合レビュー1本のみをターミナルに出力する。

### 4. 後始末（必須。正常終了か途中中止かを問わず実行する）

```sh
git worktree remove --force "$WT_DIR"
git update-ref -d refs/deep-pr-review/<PR番号>
```

worktree 作成前に中止した場合（eligibility 不適合等）は不要。エラーでフローを中断する場合も、
中断を報告する前にこの後始末を実行する。

### 出力フォーマット

```
## deep-pr-review: #<番号> <PRタイトル>

対象: <PR URL>  /  ベースブランチ: <baseRefName>
レビュー観点: 標準PRレビュー(N体) + architect メタ検証 + codex 独立レビュー
codex: <実施 / skip（理由）>
注記: <diff サイズガードで対象外にしたもの・worktree フォールバックの制約など。無ければ「なし」>
検出: BLOCKER X件 / HIGH X件 / MEDIUM X件 / LOW X件

---

### 🔴 BLOCKER
1. <一行要約>
   - 場所: `path/to/file.ext:120-130`
   - 出所: review / codex / architect（該当するものを併記）・確信度: NN
```

## 確信度ルーブリック（フェーズ3の architect に逐語で渡す）

- **0**: 全く確信なし。軽い精査で崩れる偽陽性、または既存からの問題。
- **25**: ある程度。実問題かもしれないが偽陽性の可能性もあり、検証しきれていない。様式の問題なら関連 CLAUDE.md に明記されていないもの。
- **50**: 中程度。実問題と確認できたが、nitpick か発生頻度が低い。PR 全体の中では重要度は高くない。
- **75**: 高い。再確認の上、実運用で踏む実問題である可能性が非常に高い。現状の実装は不十分。または関連 CLAUDE.md に直接記載がある問題。
- **100**: 確実。実問題で頻繁に発生すると確認済み。証拠が直接それを裏付ける。

## 偽陽性として扱う例（レビュー N 体とフェーズ3の architect に逐語で渡す。codex には要点をプロンプトに埋め込み済み）

- 既存（PR 変更前）から存在する問題。
- バグに見えるが実際はバグでないもの。
- シニアエンジニアが指摘しない過度な nitpick。
- linter / typechecker / compiler が拾う類（import 漏れ・型エラー・フォーマット）。これらは CI が別途実行する前提で、自分でビルド/型チェックはしない。
- 一般的なコード品質（テスト網羅・ドキュメント不足等）。ただし CLAUDE.md が明示要求している場合は除く。
- CLAUDE.md 記載だがコード側で明示的に抑制されている（lint ignore 等）もの。
- 意図的・本変更に直接関連すると見られる挙動変更。
- 実問題だが、ユーザーが PR で変更していない行に対するもの。
