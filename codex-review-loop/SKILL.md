# codex-review-loop — codex レビューを GREEN までループする品質ゲート

「codex レビュー → 指摘修正 → 再レビュー」を GREEN になるまで繰り返す。1 ラウンド = レビュー 1 回＋修正 1 巡。

## パラメータ

- **対象**: 引数で指定されたファイル / PR / 差分。省略時は現在の作業ツリー差分（`git diff` ＋ untracked）。PR は番号または URL を受け、`gh pr diff` で差分を取得する。対象が特定できない場合は AskUserQuestion で確認する。
- **最大ラウンド数**: `--max-rounds`（既定 5）。上限到達は失敗ではなく「人間の判断が要る」シグナルであり、深く回したい場合のみ明示的に大きい値を指定する。
- **GREEN 条件**: `--green` で指定した severity 以上の指摘がゼロになること（既定 `high`: critical / high がゼロで GREEN。`--green all` は全指摘解消）。

## codex 実行プロトコル

codex の呼び出しは必ず以下の形で行う:

```sh
TIMEOUT=$(command -v gtimeout || command -v timeout) || { echo "timeout 未導入 (macOS は brew install coreutils)"; exit 127; }
"$TIMEOUT" --kill-after=10s 300 codex exec --sandbox read-only -o <out> "<レビュー依頼プロンプト>" < /dev/null
```

- **`--sandbox read-only`（必須）**: codex のシェル実行を書込不能 sandbox に固定する。書込事故の防止と、非対話実行での承認エスカレーション待ち回避のため。書込可能 sandbox（`workspace-write` 等）は使わない。
- **`-o <out>`（必須）**: 最終メッセージをファイルに書き出す（`--output-last-message`）。stdout をパイプで受けず、必ずこのファイルを読む（パイプは無出力ハングに見える）。出力ファイルを読まずに「codex レビュー済み」としない（fire-and-forget 禁止）。
- **タイムアウト（必須・最大 300 秒）**: codex はまれにハングする（モデル応答待ち・stdin 待ち・接続スタック等）。GNU coreutils の `timeout` を使用（macOS は `gtimeout`、Linux は `timeout`。上記スニペットで自動検出）。`--kill-after=10s` は SIGTERM で死なない場合の SIGKILL 保険。終了コード `124` = タイムアウト。
- **`< /dev/null`**: 非対話実行での stdin 待ち・混入を避ける防御。
- **待機規律**: コールドスタートで 1〜2 分は正常。300 秒までは、ハング即断・ポーリング・自己中断をせず、プロセス終了か `<out>` 生成を待つ。
- 対象がリポジトリに属する場合はプロジェクトルートを cwd として実行する（cwd が異なるなら `-C <projectdir>` を付ける）。codex がレビュー中に関連コードを read-only で参照できるようにするため。

### フォールバック

1. **起動失敗**（codex 未インストール / 未ログイン / タイムアウト由来でない非 0 終了）→ codex CLI の導入・ログインをユーザーに促しつつ、そのラウンドは Claude 自身が同一観点で代替評価する。
2. **タイムアウト**（終了コード `124` / `<out>` 未生成 or 空）→ `<out>` を削除してから同一コマンドを **1 回だけリトライ**する（初回のみのハングが多い）。
3. **リトライも失敗** → Claude 自身が同一観点で代替評価する。

Claude 代替となったラウンドは、ラウンド報告と最終報告の両方に「Claude 代替（N 回試行）」と明記する（自前評価を codex 済みと偽らない）。

## git 規律

git リポジトリでは、ループは git 履歴に一切触れない。修正は作業ツリー上に累積させ、コミット・stash・ブランチ作成をしない。コミットは GREEN（または妥協終了）後、ユーザーの指示があった場合のみ行う。

## ループ

### Step 1: 対象の確定

レビュー対象を絶対パスで用意する。差分の場合は一時ファイルに書き出し、各ラウンドの開始時に `git diff`（＋未追跡ファイル）を取り直して上書きする。差分は常に HEAD 比なので、修正はコミットしなくても次ラウンドの差分に自動的に反映される。

**PR 対象の場合**: 初回のみ `gh pr diff` で取得する。修正には PR ブランチのチェックアウトが前提（未チェックアウトなら修正に進まず、Round 1 のレビュー結果を提示して終了する）。修正を行った以降のラウンドは、base ブランチとの差分（`git diff <baseRef>` ＋未追跡ファイル）に切り替えて取り直す — `gh pr diff` はローカルの未 push 修正を含まないため。

**git が無い場合**（`git rev-parse --git-dir` が失敗）:

- 対象省略時は差分既定が成立しないため、AskUserQuestion で対象を確認する。ディレクトリ全体を黙って対象にしない。
- ループ開始前に対象ファイルを一時領域へバックアップし、そのパスを報告する（修正の revert 手段の確保）。

完了条件: codex に渡す絶対パスのファイルが存在する。

### Step 2: codex レビュー

critic 観点で実行する。レビュー依頼プロンプトには以下を必ず含める:

- レビュー対象の絶対パスと種別（作業差分 / PR 差分 / 文書）。差分の場合は「これは差分ファイルであり、必要に応じてリポジトリ内の関連コードを read-only で調査してよい」と伝える
- 観点: 前提の誤り・検証不足・反証・見落としたエッジケース。read-only で批評せよ、編集・書き込みは禁止
- 指摘は `severity(critical|high|medium|low) / 箇所 / 内容 / 推奨修正` の形式で列挙せよ
- 指摘が一件もなければ本文冒頭に `GREEN` とだけ書け
- 前ラウンドで却下した指摘があれば、その却下理由（再指摘の防止）

完了条件: `<out>` ファイルを読み、指摘を severity 別に分類し終えている。

### Step 3: 判定と報告

ラウンドごとに 1 行のスコアボードを報告する: `Round N: critical X / high Y / medium Z / low W（前ラウンドから修正 M 件）`。却下理由・前ラウンドの指摘一覧などラウンド間の記録は一時領域のメモファイルに保持する（リポジトリ内にファイルを作らない）。

- GREEN 条件を満たす → Step 5 へ
- 最大ラウンド到達 → Step 5 へ（妥協終了）
- 同一の指摘（同じ箇所 file:line かつ同趣旨）が 2 ラウンド連続で解消していない → ループを止め、その指摘と対応方針をユーザーに相談する（暴走防止）
- それ以外 → Step 4 へ

### Step 4: 修正

GREEN 条件に関わる指摘から順に修正する。指摘が誤り（codex の事実誤認・プロジェクト方針との不一致）と判断した場合は、修正せず却下理由を記録する。修正後、プロジェクトの検証コマンド（test / lint 等）を実行して緑を確認し、Step 1 の対象を更新して Step 2 へ戻る。検証コマンドが該当しない対象（設計書等のドキュメント）では検証をスキップし、最終報告にその旨を明記する。

### Step 5: 最終報告

以下を含めて報告し、ループを終える:

- 総ラウンド数と最終スコアボード
- GREEN か妥協終了かの別。妥協終了の場合は未解消指摘の一覧と対応方針案（妥協案）
- Claude 代替が発生したラウンド
- 却下した指摘と理由
